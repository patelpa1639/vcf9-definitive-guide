1.1 VCF Fleet / Instance / Workload Domain Hierarchy

VMware Cloud Foundation 9.0 organizes infrastructure into a three-tier logical hierarchy. Understanding this hierarchy is fundamental to every exam objective and is the lens through which all VCF administration tasks are viewed.

Fleet

A Fleet is the highest-level organizational construct in VCF 9.0. It represents the entirety of an organization's VCF deployment and can span multiple data centers, geographies, and availability zones. The fleet provides:

  • Centralized lifecycle management — A single control plane for managing software versions, compliance, and health across all instances
  • Consistent Bill of Materials (BOM) enforcement — Ensuring all instances run compatible, validated component versions
  • Fleet-level health monitoring — Aggregated dashboards showing the status of every instance and domain
  • Cross-instance workload mobility — The ability to migrate workloads between instances using VCF Operations

Key Concept: Fleet vs. Instance

Think of a Fleet as the "enterprise view" and an Instance as the "data center view." A small organization might have a single-instance fleet, while a global enterprise could have dozens of instances under one fleet.

Instance

A VCF Instance is an autonomous deployment of VCF components within a fleet. Each instance contains exactly one Management Domain and zero or more Workload Domains. An instance is self-sufficient — it has its own vCenter Server, NSX Manager cluster, and (typically) vSAN storage. Key characteristics:

  • Each instance is deployed and managed independently but reports to the fleet
  • An instance maps to a physical or logical boundary such as a data center or region
  • The Management Domain is always the first domain deployed in a new instance
  • Instance-level operations include host commissioning, domain creation, and lifecycle updates

Workload Domain

Workload Domains are logical groupings of compute, networking, and storage resources within an instance. They are the primary unit of resource isolation and management. Each workload domain contains:

  • ESXi hosts organized into one or more vSphere clusters
  • A dedicated vCenter Server (or shared with the Management Domain in specific configurations)
  • NSX networking — either shared centralized connectivity or dedicated distributed connectivity
  • Storage — vSAN, NFS, FC, or iSCSI depending on configuration

1.2 VCF 9.0 Architectural Changes

VCF 9.0 represents the most significant architectural overhaul since the platform's inception. Understanding what changed — and what the new components replace — is critical for the exam.

VCF Installer Replaces Cloud Builder

In previous VCF versions, Cloud Builder was a dedicated OVA appliance deployed as a full virtual machine. It handled the bring-up of the Management Domain and was decommissioned afterward. In VCF 9.0:

  • VCF Installer runs as a container, not a separate VM appliance
  • It is more lightweight, faster to deploy, and easier to manage
  • The functionality remains similar: parameter validation, infrastructure deployment, and Management Domain bring-up
  • After bring-up, VCF Operations takes over day-2 management

Important: If an exam question references "Cloud Builder" in a VCF 9.0 context, it is either a distractor or testing whether you know it has been replaced. The correct answer for VCF 9.0 deployment operations is always VCF Installer.

VCF Operations Replaces SDDC Manager

VCF Operations is the new central management plane, replacing both SDDC Manager and VMware Aria Operations. It provides:

  • Fleet-level and instance-level management in a unified interface
  • Lifecycle management (upgrades, patching, BOM enforcement)
  • Health monitoring and configuration drift detection
  • Workload mobility and capacity analytics
  • Certificate and password lifecycle management

VCF Automation Replaces vRealize Automation

The self-service provisioning and infrastructure-as-code capabilities previously provided by vRealize Automation (vRA) are now delivered through VCF Automation. This includes catalog management, blueprints, ABX actions, and cloud template design.

VCF Import for Brownfield Environments

VCF Import is a new capability that allows organizations to bring existing vSphere and NSX infrastructure under VCF management without rebuilding. This "converge" path converts pre-existing environments into VCF-managed workload domains.

1.3 Fleet Management Capabilities

Fleet management in VCF 9.0 provides enterprise-wide governance and operational efficiency. Key capabilities include:

Centralized Lifecycle Management

From the fleet level, administrators can orchestrate upgrades across all instances simultaneously or in controlled waves. The BOM ensures that all component versions (ESXi, vCenter, NSX, vSAN) are validated for compatibility before any upgrade proceeds.

Fleet-Level Health Monitoring

VCF Operations aggregates health data from every instance and domain, providing:

  • Unified dashboards showing fleet-wide status at a glance
  • Proactive alerts for NTP drift, DNS misconfiguration, certificate expiry, and capacity thresholds
  • Diagnostic findings with guided remediation steps

Cross-Instance Workload Mobility

VCF Operations enables workload migration between instances within a fleet. This supports disaster recovery scenarios, data center consolidation, and geographic load balancing. Workloads can be moved using:

  • HCX (Hybrid Cloud Extension) for bulk migrations
  • vMotion for live migration within network-connected instances
  • Cross-vCenter vMotion for migrations between vCenter domains

1.4 vSphere Supervisor Deployment Modes

The vSphere Supervisor enables Kubernetes workloads on VCF infrastructure. VCF 9.0 supports four deployment modes, each designed for specific operational requirements. Knowing when to recommend each mode is a key exam skill.

Single-Zone Supervisor

The standard production deployment for most environments.

  • Requires 3 or more ESXi hosts in the cluster
  • Deploys 3 Supervisor control plane VMs for high availability
  • All control plane VMs reside in a single vSphere cluster (one failure domain)
  • Supports namespaces, Tanzu Kubernetes Grid clusters, and VM Service
  • Suitable for single-site deployments where cross-AZ resilience is not required

Multi-Zone Supervisor

Designed for maximum resilience across failure domains.

  • Stretches Supervisor control plane across multiple vSphere Zones (availability zones)
  • Each zone maps to a separate vSphere cluster, ideally in different failure domains
  • Control plane VMs are distributed across zones (one per zone with 3 zones)
  • Provides resilience against entire zone failures
  • Requires careful network design to ensure low-latency connectivity between zones

Single Host Supervisor

  • Runs the entire Supervisor on a single ESXi host
  • Designed exclusively for development and testing — NOT production
  • No high availability — host failure means complete Supervisor outage
  • Useful for proof-of-concept, training labs, and developer workstations
  • Minimal resource requirements

Important: On the exam, if a scenario describes a production workload and Single Host Supervisor is an answer choice, it is always wrong. This mode provides zero redundancy.

Easy Supervisor

  • Simplified deployment with fewer prerequisites and reduced configuration steps
  • Automates many of the networking and storage configuration steps
  • Ideal for organizations new to Kubernetes on vSphere
  • Still provides production-grade capabilities when deployed on adequate infrastructure
  • Introduced in recent VCF releases to lower the barrier to Supervisor adoption

1.5 VPC in vCenter

Virtual Private Cloud (VPC) in vCenter is a networking construct that provides workload isolation and self-service networking capabilities for tenants within VCF. It integrates tightly with NSX to deliver:

Workload Isolation

Each VPC creates a logically isolated network environment with its own address space, security policies, and routing. Tenants cannot access resources in other VPCs unless explicitly configured.

Self-Service Networking

Authorized users can create and manage their own network segments, load balancers, and firewall rules within their VPC boundary without requiring infrastructure administrator intervention. This supports:

  • Developer self-service for application networking
  • Multi-tenant environments with delegated administration
  • Compliance boundaries between business units or customers

NSX Integration

VPC networking is implemented through NSX overlay networks, Tier-1 gateways, and distributed firewall rules. The VPC abstraction simplifies what would otherwise require deep NSX knowledge.

1.6 NSX Transit Gateway: Centralized vs. Distributed

NSX Transit Gateway connectivity defines how workload domains connect to the physical network and to each other. This is one of the most important architectural decisions in VCF design.

Centralized Connectivity

  • A single Tier-0 gateway is shared across multiple workload domains
  • All north-south traffic (to/from physical network) flows through a central Edge cluster
  • Requires fewer Edge nodes overall — reduces hardware footprint
  • Simpler to manage and operate
  • Best suited for smaller deployments or environments where workload domains don't need independent routing
  • Workload domains connect via Tier-1 gateways linked to the shared Tier-0

Distributed Connectivity

  • Each workload domain gets its own Tier-0 and Tier-1 gateway pair
  • Requires Edge nodes per workload domain — higher hardware cost
  • Provides better isolation — routing failures in one domain don't affect others
  • Better performance for east-west traffic between workloads within the same domain
  • Required for large-scale, multi-tenant deployments
  • Enables independent routing policies and firewall rules per domain

Key Concept: When Are Edge Nodes Required?

Edge nodes are required when north-south routing is needed (traffic leaving the NSX overlay). In centralized mode, only the central Edge cluster handles this. In distributed mode, each workload domain needs its own Edge nodes. East-west traffic within the overlay does not require Edge nodes — it is handled by the distributed router on each host.

2.1 Three VCF Deployment Paths

VCF 9.0 supports three distinct deployment paths, each designed for a specific starting point. Choosing the correct path is often tested on the exam.

Path 1: New Fleet (Greenfield)

A completely fresh installation with no prior VMware infrastructure. This is the most common deployment scenario for new environments.

  • Deploy VCF Installer container on a temporary bootstrap machine
  • Complete the parameter workbook with all required infrastructure details
  • VCF Installer validates prerequisites and deploys the Management Domain
  • This creates the first Instance within the new Fleet
  • After Management Domain bring-up, create Workload Domains as needed

Path 2: New Instance in Existing Fleet

Adding a new VCF Instance to an existing Fleet — for example, deploying in a second data center or region.

  • The existing Fleet provides the management context
  • VCF Installer deploys a new Management Domain for the new Instance
  • The new Instance registers with the Fleet for centralized management
  • Useful for geographic expansion, DR site deployment, or capacity scaling

Path 3: Converge Existing Infrastructure (Brownfield)

Using VCF Import to bring existing vSphere and NSX infrastructure under VCF management without rebuilding.

  • Requires compatible ESXi and vCenter versions (must meet VCF 9.0 BOM requirements)
  • NSX must already be deployed on the target infrastructure — this is a hard prerequisite
  • The import process converts existing infrastructure into a VCF-managed Workload Domain
  • Preserves existing VMs and configurations
  • Validates storage, networking, and compute compatibility before proceeding

Key Concept: Greenfield vs. Brownfield

Greenfield = new deployment, no existing infrastructure (Paths 1 & 2). Brownfield = existing infrastructure being converted (Path 3 / VCF Import). The exam frequently uses "given a scenario" to test which path is appropriate.

2.2 Management Domain Bring-Up

The Management Domain is always the first domain deployed in any VCF Instance. It hosts the management components that control all other domains.

Prerequisites

Before starting bring-up, the following must be in place:

  • DNS — Forward AND reverse records for all management components (vCenter, NSX Managers, ESXi hosts, VCF Operations). Both directions must resolve correctly.
  • NTP — A reliable NTP source accessible to all components. Time synchronization failures are the #1 cause of deployment issues.
  • Networking — VLANs configured for management, vMotion, vSAN, and overlay traffic. Physical switch port configuration must be complete.
  • IP Pools — IP address ranges defined for TEP (Tunnel Endpoint), host management, and other overlay requirements.
  • ESXi Hosts — Minimum 4 hosts for vSAN-based Management Domain (3 for the cluster + headroom), or 2 hosts for non-vSAN storage.

Parameter Workbook

The deployment parameter workbook (Excel spreadsheet or JSON) contains every configuration value needed for bring-up:

  • Management network CIDR, gateway, VLAN ID
  • vMotion network CIDR, gateway, VLAN ID
  • vSAN network CIDR, gateway, VLAN ID
  • DNS server addresses, NTP server addresses, domain name
  • ESXi host IPs, credentials, and hostnames
  • vCenter FQDN, IP, and root password
  • NSX Manager cluster FQDN, VIP, and individual node IPs
  • License keys (applied post-deployment)

Deployment Stages

  1. Validation — VCF Installer checks all prerequisites: DNS resolution, NTP connectivity, network reachability, host accessibility, password complexity
  2. ESXi Configuration — Hosts are imaged (if needed) and configured per the workbook
  3. vCenter Deployment — vCenter Server Appliance is deployed on the first host
  4. Cluster Creation — ESXi hosts are added to a vSphere cluster with HA and DRS enabled
  5. vSAN Configuration — vSAN datastore is created across cluster hosts (if using vSAN)
  6. NSX Deployment — NSX Manager cluster (3 nodes) is deployed and configured
  7. VCF Operations Deployment — The management plane is deployed and configured
  8. Validation — Post-deployment health checks confirm all components are operational

2.3 Host Commissioning and Decommissioning

Commissioning Prerequisites

Before a host can be added to VCF (commissioned), it must meet these requirements:

  • ESXi version must match the current VCF BOM — Mismatched versions will fail validation
  • Network configuration — Management VLAN trunked to the host's physical NICs
  • DNS records — Forward AND reverse records must exist and resolve correctly
  • NTP configured — Host must be syncing with the same NTP source as other VCF components
  • SSH enabled — Required for VCF to configure the host during commissioning
  • Default credentials — ESXi root password must be known to VCF
  • No existing vCenter membership — The host must not already be managed by another vCenter

Decommissioning Process

Removing a host from VCF requires careful preparation:

  1. Evacuate workloads — All VMs must be migrated off the host (vMotion or power off)
  2. Enter maintenance mode — The host must be in maintenance mode before decommissioning
  3. vSAN data evacuation — If using vSAN, data must be evacuated. Choose "Evacuate all data" for full decommission. "Ensure accessibility" means some data may be at risk.
  4. Remove from cluster — The host is removed from the vSphere cluster
  5. Decommission in VCF — VCF Operations removes the host from its inventory and reclaims resources

Important: You cannot decommission a host if doing so would violate the minimum host count for the cluster (3 for vSAN clusters, 2 for non-vSAN). The operation will fail with a validation error.

2.4 Scaling Clusters and Domains

Adding Hosts to Existing Clusters

The most common scaling operation. Commission new hosts, then add them to an existing cluster through VCF Operations. The hosts are automatically configured with the cluster's networking, storage, and NSX transport node profiles.

Adding Clusters to Workload Domains

When a single cluster isn't sufficient, add a new cluster within the existing workload domain. Specify the cluster configuration, host assignment, storage type, and networking profile.

Creating New Workload Domains

For workload isolation or different configuration requirements, create an entirely new workload domain. This deploys a new vCenter Server and configures NSX connectivity (centralized or distributed).

2.5 Importing vCenter as Workload Domain (VCF Import)

VCF Import converts existing vSphere infrastructure into a VCF-managed workload domain. This is the brownfield path for organizations that want VCF management without rebuilding.

Requirements

  • ESXi and vCenter versions must be compatible with VCF 9.0 BOM
  • NSX must already be deployed on the target environment — VCF Import does not install NSX
  • Storage must be vSAN or a supported external storage type
  • Hosts must meet VCF hardware compatibility requirements
  • Network configuration must align with VCF standards (appropriate VLANs, MTU settings)

Import Process

  1. Run the VCF Import validation tool against the target environment
  2. Resolve any compatibility issues identified in the validation report
  3. Initiate the import through VCF Operations
  4. VCF converts the existing vCenter and its clusters into a managed Workload Domain
  5. All existing VMs and configurations are preserved
  6. The imported domain appears in the VCF inventory alongside other managed domains

2.6 Post-Deployment Checklist

After successful deployment, several critical tasks must be completed immediately. The order matters — and the exam tests this.

Critical: Licensing must be applied IMMEDIATELY after deployment. This is always the correct answer to "what is the first step after deployment?" questions on the exam.

Post-Deployment Tasks (in order)

  1. Apply Licenses — Immediately. Components run in evaluation mode until licensed. Evaluation mode expires and stops workloads.
  2. Configure Backup — Set up backup for vCenter, NSX Manager, and VCF Operations. File-based backup to an external target.
  3. Verify Health Checks — Run VCF Health to confirm all components pass validation
  4. Configure Identity Sources — Integrate with Active Directory or LDAP for centralized authentication
  5. Configure Syslog — Forward logs to a central log management solution
  6. Review Certificate Configuration — Verify certificates are properly signed and not using defaults

2.7 Air-Gapped Deployments

For environments without internet access (government, financial, secure facilities), VCF supports air-gapped deployment using the VCF Download Tool.

Process

  1. Connected System — Use the VCF Download Tool on an internet-connected machine to download all required bundles (ESXi images, vCenter, NSX, patches)
  2. Transfer — Move the downloaded bundles to the air-gapped environment via physical media or secure transfer
  3. Configure Local Depot — Set up a local depot server within the air-gapped environment
  4. Point VCF to Local Depot — Configure VCF Operations to use the local depot instead of VMware's online repository
  5. Proceed with Normal Deployment — The bring-up and lifecycle processes work identically, pulling from the local depot

Key Concept: Air-Gapped Lifecycle

Air-gapped environments require ongoing bundle downloads and transfers for every patch and upgrade cycle. The VCF Download Tool must be used before each update to stage the required bundles in the local depot.

3.1 NSX in VCF 9.0 — No Standalone Upgrade

This is one of the most important changes in VCF 9.0 and is heavily tested on the exam.

Critical Rule: In VCF 9.0, you CANNOT upgrade NSX independently. All NSX upgrades must be performed through the VCF lifecycle management process using the validated Bill of Materials (BOM).

In previous VCF versions, administrators could sometimes upgrade NSX Manager and Transport Nodes independently using the NSX upgrade coordinator. VCF 9.0 eliminates this option to ensure:

  • Component compatibility — The BOM guarantees that all component versions (ESXi, vCenter, NSX, vSAN) are tested and validated together
  • Upgrade sequencing — The correct upgrade order is automatically enforced (NSX before vCenter before ESXi)
  • Rollback safety — VCF lifecycle management includes pre-checks and rollback capabilities
  • Fleet consistency — All instances in the fleet maintain compatible versions

What This Means on the Exam

If a question asks "How should an administrator upgrade NSX in a VCF 9.0 environment?" — the answer is always "Use VCF lifecycle management" or "Apply the update through VCF Operations using the validated BOM." Any answer suggesting direct NSX upgrade, standalone NSX upgrade coordinator, or manual NSX binary replacement is wrong.

3.2 Centralized vs. Distributed Connectivity

The NSX connectivity model determines how workload domains connect to the physical network and to each other. This decision is made during workload domain creation and has significant implications for architecture, performance, and cost.

Centralized Connectivity

In the centralized model, a single shared Tier-0 gateway serves multiple workload domains. This gateway runs on a dedicated Edge cluster (typically in the Management Domain or a shared infrastructure domain).

Traffic flow:

  • North-south traffic from any workload domain exits through the central Tier-0 Edge cluster
  • Each workload domain connects via a dedicated Tier-1 gateway linked to the shared Tier-0
  • East-west traffic between VMs in the same domain stays on the distributed router (no Edge traversal)
  • East-west traffic between different workload domains traverses the central Tier-0

When to use:

  • Smaller deployments (fewer than 5 workload domains)
  • Environments where cost reduction is a priority
  • Workloads that don't require independent routing policies
  • When simplified operations outweigh the need for per-domain isolation

Distributed Connectivity

In the distributed model, each workload domain gets its own Tier-0 and Tier-1 gateways and its own Edge cluster.

Traffic flow:

  • North-south traffic exits through the domain's own Edge cluster
  • Each domain has independent routing tables, BGP peering, and firewall rules
  • Routing failures in one domain do not affect other domains
  • Each domain can have different uplink connectivity (different VLANs, different physical routers)

When to use:

  • Large-scale multi-tenant deployments
  • When workload domains require independent routing policies
  • Compliance requirements demanding network isolation between domains
  • When per-domain performance guarantees are needed

3.3 Gateway Firewall Behavior in NSX 9.0

This is a critical behavioral change that the exam tests directly.

NSX 9.0 Change: In greenfield (new) NSX 9.0 deployments, the Gateway Firewall is DISABLED by default on both Tier-0 and Tier-1 gateways.

What Changed

  • Previous versions: Gateway Firewall was enabled by default on Tier-0 and Tier-1 gateways with a default allow-all rule
  • NSX 9.0 greenfield: Gateway Firewall is completely disabled. No gateway firewall rules are evaluated. Traffic passes through gateways without any firewall inspection at the gateway level.
  • NSX 9.0 brownfield (upgraded): Existing gateway firewall configuration is preserved. The change only affects new deployments.

Implications

  • If you need gateway-level firewall inspection (north-south filtering at the T0 or T1), you must explicitly enable Gateway Firewall after deployment
  • The Distributed Firewall (DFW) for east-west micro-segmentation is unaffected by this change
  • This reduces the default processing overhead on gateway traffic for environments that don't need gateway firewalling

Troubleshooting Scenario

If an administrator reports that gateway firewall rules are not being applied in a new VCF 9.0 deployment, the first thing to check is whether Gateway Firewall has been enabled on the relevant T0/T1 gateway. This is the correct answer in a "given a scenario" question.

3.4 Edge Host Affinity

Edge Host Affinity is a critical feature for maintaining network connectivity during infrastructure maintenance operations, particularly ESXi host upgrades.

The Problem It Solves

When an ESXi host running Edge VMs needs to enter maintenance mode for an upgrade:

  • Edge VMs must be moved off the host
  • vMotion of Edge VMs can cause network disruption (brief traffic loss during migration)
  • In some configurations, Edge VMs cannot be vMotioned (e.g., SR-IOV passthrough, specific DPDK configurations)

How Edge Host Affinity Works

  1. Define affinity rules — Specify which hosts are designated failover targets for Edge VMs
  2. During maintenance — When a host enters maintenance mode, Edge VMs are powered off and restarted on the designated affinity hosts (instead of vMotioned)
  3. Controlled failover — The NSX Edge cluster handles the failover automatically. The standby Edge takes over active services while the primary Edge is restarted on the new host.
  4. After maintenance — Edge VMs can be moved back to their original hosts

Key Concept: Failover Without vMotion

Edge Host Affinity enables Edge VM failover without requiring vMotion. The Edge VM is shut down on the source host and restarted on the target host. NSX Edge HA handles the service failover. This is fundamentally different from DRS-based VM migration.

When It Matters

  • ESXi upgrades — Hosts must be evacuated and rebooted; Edge Host Affinity ensures Edges restart on designated hosts
  • Hardware maintenance — Planned maintenance windows for server hardware
  • Host failures — HA restarts Edge VMs on affinity-designated hosts for predictable recovery

3.5 CARR Script and Certificate Management

The CARR (Certificate Authority Renewal and Rotation) script is a VCF tool for managing the lifecycle of certificates used by NSX components, particularly Transport Node (TN) certificates.

The 825-Day Threshold

Critical Detail: Transport Node certificates in NSX have an 825-day validity period. When certificates approach this threshold, they must be renewed to maintain secure communication between NSX Manager and transport nodes.

What Happens If Certificates Expire

  • NSX Manager loses the ability to communicate securely with Transport Nodes
  • Transport Node configuration changes cannot be pushed
  • New overlay segments cannot be provisioned on affected hosts
  • Health monitoring data from Transport Nodes stops flowing to NSX Manager
  • In severe cases, existing overlay networking may be disrupted

CARR Script Operations

  • Certificate status check — Report on certificate validity, expiration dates, and renewal status across all transport nodes
  • Certificate renewal — Automated renewal of expiring certificates without service disruption
  • Certificate rotation — Replace certificates with new ones (e.g., when migrating to a new CA)
  • Bulk operations — Process all transport nodes in a cluster or domain simultaneously

Best Practice

Monitor certificate expiration proactively using VCF Health checks. Schedule CARR script runs well before the 825-day threshold. VCF Health will alert when certificates are within 30 days of expiration.

3.6 NSX Networking Design Requirements

MTU Configuration

  • Minimum MTU for overlay networks: 1600 bytes — Geneve encapsulation adds overhead to packets
  • Recommended MTU: 9000 (jumbo frames) for best performance
  • MTU must be configured consistently on physical switches, ESXi vSwitches, and any intermediate network devices
  • MTU mismatch is a common cause of intermittent connectivity issues and packet drops

VLAN Trunk Configuration

Physical switch ports connecting to ESXi hosts must trunk the following VLANs:

  • Management VLAN
  • vMotion VLAN
  • vSAN VLAN (if using vSAN)
  • TEP (Overlay) VLAN — for Geneve tunnel endpoints
  • Edge uplink VLAN(s) — for NSX Edge connectivity to physical network

TEP (Tunnel Endpoint) IP Pools

  • Each ESXi host configured as a Transport Node requires at least one TEP IP address
  • TEP IPs are assigned from IP pools defined in NSX Manager
  • TEP pools must be large enough to accommodate all current and planned hosts
  • TEP communication uses the overlay VLAN and requires L2 adjacency or proper L3 routing

BGP and Routing for Tier-0 Uplinks

  • Tier-0 gateways typically peer with physical routers using BGP
  • Static routing is an alternative for simpler environments
  • ECMP (Equal-Cost Multi-Path) routing can be used for load balancing across multiple Edge uplinks
  • BFD (Bidirectional Forwarding Detection) provides fast failure detection for BGP sessions

4.1 vSAN ESA vs. OSA

VCF 9.0 supports two vSAN storage architectures. Understanding the differences is critical — the exam tests this extensively.

ESA — Express Storage Architecture

ESA is the next-generation vSAN architecture and the default for new VCF 9.0 deployments.

  • Single storage tier — No separate cache and capacity tiers. All drives participate in a single, flat storage pool.
  • NVMe-only — Requires all NVMe flash drives. No support for SAS, SATA SSD, or HDD.
  • Improved erasure coding — RAID-5 requires only 4 hosts (vs. 4+ in OSA). RAID-6 requires 6 hosts. More efficient capacity utilization.
  • Native compression and deduplication — Always enabled, no performance penalty. Built into the data path.
  • Better performance — Flat architecture eliminates the cache-to-capacity destaging overhead.
  • Simplified management — No need to size cache devices separately from capacity devices.
  • Snapshots — Improved snapshot performance with reduced overhead.

OSA — Original Storage Architecture

OSA is the traditional vSAN architecture supported for existing/legacy deployments.

  • Two-tier architecture — Separate cache tier (fast) and capacity tier (larger, slower).
  • Cache tier requirements — Minimum 1 cache device per disk group (SSD or NVMe). Cache should be ~10% of capacity tier size.
  • Capacity tier options — SSD, NVMe, or HDD (hybrid mode uses HDD capacity + SSD cache).
  • Disk groups — Storage is organized into disk groups, each with 1 cache device + 1-7 capacity devices.
  • Erasure coding — RAID-5 requires 4+ hosts; RAID-6 requires 6+ hosts.
  • Optional dedup/compression — Must be explicitly enabled; has some performance impact on all-flash configurations.

4.2 Stretched Cluster Requirements

vSAN stretched clusters span two physical sites (data sites) with a witness host in a third fault domain. This provides site-level resilience for VCF workload domains.

Witness Host Requirements

  • Must be in a third fault domain — physically and logically separate from both data sites
  • Runs as a specialized ESXi appliance (vSAN Witness Appliance)
  • Stores only witness components (metadata) — no VM data
  • Minimum specs: 2 vCPU, 8 GB RAM (scales with number of objects)
  • Requires network connectivity to both data sites
  • Can be shared across multiple stretched clusters (max 64)

Latency Requirements

Latency limits — memorize these:

  • ≤5 ms RTT between the two data sites (production network latency)
  • ≤200 ms RTT from each data site to the witness host

These are ROUND-TRIP times, not one-way. The witness has a much more relaxed latency requirement because it only handles metadata.

Availability Zone Deployment

  • In VCF, each data site maps to an Availability Zone (AZ)
  • The stretched cluster spans both AZs
  • VM placement policies can enforce site affinity — keeping specific VMs on a preferred site
  • During a site failure, VMs restart on the surviving site via vSphere HA

4.3 Storage Policies

vSAN storage policies define the protection level, performance characteristics, and data services for VMs in VCF workload domains.

Key Policy Parameters

  • FTT (Failures to Tolerate) — Number of host/disk/fault-domain failures the data can survive:
    • FTT=0: No protection (development only)
    • FTT=1: Survives 1 failure (RAID-1 mirror: 2x capacity; RAID-5 erasure coding: 1.33x capacity)
    • FTT=2: Survives 2 failures (RAID-1 triple mirror: 3x capacity; RAID-6 erasure coding: 1.5x capacity)
    • FTT=3: Survives 3 failures (RAID-1 requires 4 copies: 4x capacity)
  • Failure Tolerance Method — RAID-1 (mirroring) or RAID-5/6 (erasure coding). Erasure coding is more capacity-efficient but has slightly higher CPU overhead.
  • Stripe Width — Number of capacity devices across which each object is striped (affects performance).
  • Force Provisioning — Allow object creation even if the policy can't be fully satisfied (NOT recommended for production).
  • Encryption — Can be enforced at the policy level (requires KMS).

Default vs. Custom Policies

  • VCF deploys a default storage policy with FTT=1 and RAID-1 mirroring
  • Custom policies can be created for workload-specific requirements
  • Policies can be assigned at the namespace level (for Supervisor) or per-VM
  • Changing a policy triggers automatic data rebalancing (can be resource-intensive)

4.4 vSAN Encryption

Data-at-Rest Encryption (D@RE)

  • Encrypts all data stored on vSAN disks
  • Requires an external Key Management Server (KMS) — vSAN does not include a built-in KMS
  • Supports KMIP-compliant KMS solutions (e.g., HyTrust, Thales, Entrust)
  • Encryption occurs below the vSAN layer — transparent to VMs
  • Performance impact is minimal on modern CPUs with AES-NI acceleration
  • Can be enabled per-cluster

Data-in-Transit Encryption (DiTE)

  • Encrypts vSAN network traffic between hosts
  • Does NOT require a KMS — uses ephemeral keys generated by the hosts
  • Protects against network sniffing and man-in-the-middle attacks on the vSAN network
  • Some performance impact due to encryption/decryption of all inter-host traffic
  • Can be enabled independently of data-at-rest encryption

Key Concept: KMS Requirements

Data-at-Rest encryption requires KMS. Data-in-Transit encryption does NOT require KMS. This distinction is tested directly. If a question asks about enabling vSAN encryption without a KMS, only in-transit encryption is possible.

Combining Both

You can enable both D@RE and DiTE simultaneously for maximum protection. This is recommended for environments with strict compliance requirements (PCI-DSS, HIPAA, FedRAMP). The D@RE KMS dependency still applies.

5.2 Bill of Materials (BOM) and Upgrade Waves

The VCF Bill of Materials is the cornerstone of lifecycle management. It defines the exact combination of component versions that have been validated to work together.

What the BOM Defines

  • ESXi version and build number
  • vCenter Server version
  • NSX Manager and Edge version
  • vSAN version (tied to ESXi)
  • VCF Operations version
  • VCF Automation version
  • All other VCF component versions

Upgrade Wave Order

The upgrade order is FIXED and cannot be changed:

  1. Pre-check validation — Mandatory; must pass before any component upgrade begins
  2. NSX — Upgraded first (Manager cluster, then Edge nodes, then host transport nodes)
  3. vCenter Server — Upgraded after NSX
  4. ESXi — Hosts upgraded in rolling fashion (one at a time, with evacuation)
  5. vSAN — On-disk format upgrade (if required by the new version)

This order exists because each component may depend on features or APIs in the previously upgraded component. For example, vCenter may require NSX APIs that only exist in the new NSX version.

Pre-Check Validation

  • Validates hardware compatibility with the target BOM
  • Checks available disk space for upgrade packages
  • Verifies network connectivity between all components
  • Confirms backup status of critical components
  • Checks certificate validity (prevents upgrade from failing mid-process due to expired certs)
  • Pre-check failures MUST be resolved before proceeding — there is no "skip pre-check" option

Rollback Considerations

  • VCF takes snapshots and backups before each upgrade step
  • Rollback is possible for individual components if the upgrade fails
  • Rolling back after multiple components have been upgraded may require manual intervention
  • Always verify backup integrity before starting an upgrade

5.3 Configuration Drift Detection

  • Syslog targets — Confirms log forwarding destinations are configured and reachable
  • Certificate status — Monitors certificate validity and warns before expiration
  • Password rotation compliance — Tracks password age against rotation policies
  • Host profiles — Detects ESXi configuration deviations from the baseline profile
  • NSX transport node configuration — Verifies TN profiles match the desired configuration

Remediation Workflows

  • Guided remediation — VCF provides step-by-step instructions for issues requiring manual intervention
  • Drift reports — Exportable reports showing what changed, when, and the current vs. expected values

5.4 ESX Live Patch

ESX Live Patch allows administrators to apply certain ESXi patches without requiring a host reboot, reducing maintenance windows and workload disruption.

How It Works

  • Patches are classified as "live-patchable" or "reboot-required" by VMware
  • Live patches are applied to the running ESXi kernel in-place
  • No need to evacuate VMs (no maintenance mode required for live patches)
  • Applied through VCF lifecycle management like any other patch

Limitations

  • Not all patches qualify — Only specific security and bug-fix patches are live-patchable
  • Feature updates typically require reboot — Major ESXi updates are not live-patchable

6.1 VCF Fleet Deployment Issues (Objective 5.1)

The VCF bring-up process is an orchestrated, multi-phase deployment that stands up the entire management domain from bare metal. Because it touches DNS, NTP, networking, storage, and compute simultaneously, failures can cascade and the root cause is often in the infrastructure layer rather than VCF itself. Understanding the order in which Cloud Builder validates and deploys components is essential to diagnosing where and why a failure occurred.

Common Bring-Up Failures

DNS Resolution Failures. DNS is the single most common cause of bring-up failure. The VCF Installer (Cloud Builder) validates DNS as one of its first pre-checks, but administrators frequently configure forward records while omitting reverse (PTR) records. Every component — SDDC Manager, vCenter, NSX Manager, ESXi hosts — requires both a forward (A) and reverse (PTR) record. If the bring-up stalls at the vCenter deployment phase, your first troubleshooting step should always be to verify DNS from the Cloud Builder appliance itself:

# From Cloud Builder CLI
nslookup vcenter-mgmt.vcf.lab
nslookup <vCenter-IP>
# Both must resolve correctly and match each other

# Verify all management components
nslookup sddc-manager.vcf.lab
nslookup nsxmgr-01.vcf.lab

Exam Trap: When a deployment fails at DNS, always check BOTH forward AND reverse records. The exam will present scenarios where forward lookup works but the PTR record is missing or points to the wrong hostname. This is a classic distractor — do not assume that a successful nslookup hostname means DNS is fully configured.

NTP Synchronization Issues. VCF requires all hosts and appliances to synchronize time from the same NTP source. A time skew of more than a few seconds between ESXi hosts and the NTP server will cause certificate validation failures and SSO authentication errors during bring-up. Time skew is particularly insidious because it may not produce an obvious error message — instead, you see TLS handshake failures or token validation errors that appear to be certificate problems. Verify NTP status on ESXi hosts with:

# On each ESXi host
esxcli system ntp get
esxcli system ntp config get
# Verify time offset
esxcli system time get

Incorrect Parameter Workbook Values. The deployment parameter workbook (Excel spreadsheet) feeds Cloud Builder with every IP address, hostname, VLAN, credential, and configuration value. Common errors include overlapping IP ranges, mismatched VLAN IDs between the workbook and physical switch configuration, incorrect subnet masks, gateway IPs that do not match the network infrastructure, and passwords that do not meet complexity requirements. Always validate the workbook using the Cloud Builder pre-check validation before initiating deployment. Pay special attention to the management, vMotion, vSAN, and TEP network pool definitions — a single transposed digit in a subnet mask will cause cascading failures.

Network Connectivity Problems. Bring-up requires layer-2 and layer-3 connectivity across management, vMotion, vSAN, and TEP VLANs. If the physical switch trunk ports do not carry all required VLANs, or if MTU is not set to 9000 on vSAN and TEP networks, components will fail to communicate. Test jumbo frame connectivity from ESXi hosts before initiating bring-up:

# Test MTU 9000 connectivity for vSAN network
vmkping -s 8972 -d <remote-vSAN-IP>

# Test TEP network jumbo frames
vmkping -I vmk10 -s 8972 -d <remote-TEP-IP>

Insufficient Resources. Cloud Builder validates that each ESXi host meets minimum CPU, memory, and disk requirements. A host with fewer than the required number of disk groups for vSAN, or insufficient memory to host management VMs (vCenter, NSX Manager, SDDC Manager), will fail validation. Remember that the management domain requires significant overhead — at least 4 hosts with sufficient resources to run the full management stack.

Key Log Locations

Cloud Builder stores deployment logs in /opt/vmware/bringup/logs/. The primary log file is vcf-bringup-debug.log. For SDDC Manager post-deployment issues, check /var/log/vmware/vcf/sddc-manager-ui-app/ and /var/log/vmware/vcf/operationsmanager/. The Cloud Builder validation report is also saved locally and provides a detailed breakdown of every pre-check result.

Resuming Failed Deployments

When bring-up fails, you do not need to start from scratch. Cloud Builder supports resuming from the point of failure after you correct the underlying issue. Use the SDDC Manager UI or API to retry the failed task. If the failure is unrecoverable (for example, a corrupted appliance deployment), Cloud Builder provides a "cleanup" option that tears down partially deployed components so you can restart cleanly. Always fix the root cause before resuming — simply retrying without addressing the issue will produce the same failure.

6.2 Upgrade and Conversion Issues (Objective 5.2)

VCF lifecycle management (LCM) orchestrates upgrades across vCenter, ESXi, NSX, and vSAN in coordinated "waves." Understanding the upgrade workflow is critical because the exam tests your ability to diagnose failures at each stage and determine the correct remediation path.

Failed Upgrade Pre-Checks

Before any upgrade begins, SDDC Manager runs a series of pre-checks that validate BOM compatibility, component health, resource availability, and network connectivity. Pre-check failures must be resolved before the upgrade can proceed. Common pre-check failures include:

  • BOM Compatibility: The target version's Bill of Materials (BOM) specifies exact supported versions of each component. If your current NSX version is not on the supported upgrade path for the target VCF release, the pre-check will fail. You must apply an async patch to bring the component to a supported version first.
  • Component Health: Unhealthy vSAN clusters, degraded NSX controllers, or disconnected hosts will block upgrades. Resolve all health alarms and issues before retrying pre-checks.
  • Resource Availability: Upgrades require temporary additional resources for rolling restarts. Ensure sufficient capacity exists in the cluster to accommodate hosts entering maintenance mode during the rolling upgrade process.
  • Password Expiration: If service account passwords have expired or are about to expire, pre-checks will flag this. Rotate credentials through SDDC Manager before proceeding.

Important: The "skip pre-check" option should never be used in production and is never the correct answer on the exam. Pre-checks exist to prevent catastrophic upgrade failures. Always resolve the root cause rather than bypassing the safety mechanism.

Upgrade Wave Failures

VCF upgrades components in a specific order: NSX first, then vCenter, then ESXi hosts (rolling), then vSAN on-disk format if applicable. A failure in any wave halts the entire upgrade sequence. Common wave failures include:

  • NSX upgrade failure: Check NSX Manager cluster status and Edge cluster upgrade state. If an Edge VM fails to upgrade, verify resource availability on the target host and network connectivity. NSX Manager nodes are upgraded one at a time — if the first node fails, the remaining nodes retain the previous version.
  • ESXi rolling upgrade failure: A host may fail to enter maintenance mode if DRS cannot evacuate all VMs. Common blockers include VM-host affinity rules that prevent migration, insufficient capacity on remaining hosts, VMs with local storage that cannot vMotion, and USB passthrough devices.
  • vCenter upgrade failure: The VCSA upgrade takes a snapshot before proceeding. If the upgrade fails, you can revert to the snapshot. Check /var/log/vmware/upgrade/ for detailed upgrade logs.

Rollback Procedures

VCF supports rollback for certain upgrade stages. vCenter upgrades can be rolled back if the upgrade has not yet committed (the previous VCSA is preserved as a powered-off VM). ESXi upgrades on individual hosts can be rolled back by reinstalling the previous image via SDDC Manager. However, once vSAN on-disk format upgrades are committed, they cannot be reversed — this is a one-way operation. The key log for upgrade troubleshooting is found in SDDC Manager at /var/log/vmware/vcf/lcm/.

VCF Import and Async Patch Failures

Converting existing vSphere environments to VCF-managed (VCF Import) requires that the source environment meet strict compatibility requirements. Common import errors include unsupported hardware models, non-compliant networking configurations (non-VDS switches), version mismatches, and missing required VCF network pools. Async patches — out-of-cycle patches applied between VCF releases — can fail if the patch bundle is corrupted, if dependent patches are missing, or if the target component is in an unhealthy state. Always verify bundle integrity using the checksum provided in the download portal before uploading to SDDC Manager.

6.3 vSphere/vCenter Troubleshooting (Objective 5.3)

vCenter Server is the central management plane for the entire VCF stack. When vCenter is unhealthy, nearly every other operation is impacted — SDDC Manager cannot communicate with the infrastructure, DRS and HA cannot function, and administrators lose visibility into the environment.

vCenter Service Health

The vCenter Server Appliance (VCSA) runs dozens of interdependent services. The most critical service is VPXD (the vCenter Server daemon), which handles all inventory, task, and event management. Check service status with:

# SSH to vCenter appliance
service-control --status --all

# Check a specific service
service-control --status vpxd

# Restart a failed service
service-control --start vpxd

# View VCSA disk space (common cause of service failures)
df -h

If VPXD will not start, examine /var/log/vmware/vpxd/vpxd.log for errors. Common causes include database connectivity issues (the embedded PostgreSQL database may be full or corrupted), certificate expiration, memory exhaustion on the VCSA, and disk space exhaustion (particularly the /storage/log partition filling up with old log files).

SSO and Identity Issues

Single Sign-On (SSO) problems manifest as login failures to both the vSphere Client and SDDC Manager (since SDDC Manager authenticates through vCenter SSO). Check the SSO service (vmware-stsd) and inspect /var/log/vmware/sso/ for token issuance errors. If SSO certificates have expired, use the certificate-manager utility on the VCSA to renew them. For Active Directory integration issues, verify network connectivity to domain controllers and check that the machine account password has not expired.

Performance Degradation

vCenter performance issues typically manifest as slow UI response, task timeouts, or inventory sync delays. Check VCSA resource usage (CPU, memory, disk I/O) through the VAMI interface at https://<vcenter>:5480. Common causes include undersized VCSA deployment (too many hosts for the selected size), excessive logging, and database bloat from unresolved alarms or orphaned tasks.

6.4 ESXi and HA/DRS Issues (Objective 5.4)

Host Disconnected States

An ESXi host showing as "Disconnected" or "Not Responding" in vCenter can be caused by management network failure, hostd service crash, vpxa agent failure, or a complete host lockup. Troubleshoot by attempting direct access through SSH or the DCUI (Direct Console User Interface). If the host is reachable via SSH, restart the management agents:

# On the ESXi host — restart management agents
/etc/init.d/hostd restart
/etc/init.d/vpxa restart

# Alternative: restart all management services
services.sh restart

The primary host log is /var/log/hostd.log. For management agent communication issues, also check /var/log/vpxa.log. If the host is completely unresponsive, check physical hardware health through the out-of-band management interface (iLO, iDRAC, IMM).

HA Failover Problems

vSphere HA uses the FDM (Fault Domain Manager) agent on each host. When HA detects a host failure, it restarts protected VMs on surviving hosts. Common HA issues tested on the exam include:

  • Isolation Response: When a host loses all network heartbeats, it checks heartbeat datastores before declaring itself isolated. If heartbeat datastores are configured and responding, the host determines it is in a network partition (not truly isolated) and VMs continue running. The configured isolation response (leave powered on, power off, or shut down) only triggers after both network and datastore heartbeats fail. This distinction is critical for exam questions.
  • Admission Control: If admission control is configured with a percentage-based policy or a dedicated failover host, HA may refuse to power on VMs if doing so would violate the failover capacity reservation. This appears as a "not enough resources" error even when physical capacity appears available. The reserved capacity is set aside for failover and cannot be consumed by normal operations.
  • Split-Brain Resolution: A split-brain condition occurs when a host is network-isolated but its VMs are still running, and HA simultaneously restarts those VMs on another host. Two copies of the same VM briefly exist. vSphere resolves this by using VMDK lock files on shared storage — the original VM retains the lock, and the duplicate is powered off automatically.
  • Master Election: HA elects a master host per cluster that is responsible for monitoring all slave hosts and initiating VM restarts. If the master host fails, a new election occurs. If you see HA errors about "no eligible master," check that the hosts can communicate on the management network and that the HA agent is running.

Key Concept: HA Heartbeat Decision Flow

Network heartbeats fail → Check heartbeat datastores → If datastore heartbeats also fail → Host is truly isolated → Execute configured isolation response (power off, shut down, or leave powered on). If datastore heartbeats succeed → Host network is partitioned but not fully isolated → VMs continue running on the isolated host, and HA does NOT restart them elsewhere.

The HA agent log is /var/log/fdm.log on each ESXi host. This log contains all election decisions, failure detection events, and VM restart actions. For DRS-related issues, check vpxd.log on vCenter, as DRS is a vCenter-level function.

DRS Imbalance and Host Profile Compliance

DRS imbalance occurs when the automation level is too conservative (set to "Manual" or "Partially Automated"), when VM-host affinity rules prevent optimal placement, or when resource reservations create artificial constraints. Check DRS faults and recommendations in the vCenter cluster summary tab. Remember that DRS operates on a 5-minute cycle by default — changes are not instantaneous.

Host profile compliance failures indicate configuration drift between the host's running configuration and the reference profile. Common drift areas include NTP settings, syslog configuration, DNS settings, and security policies. Use the "Check Compliance" action to identify specific deviations, then "Remediate" to bring hosts back into compliance. Be aware that remediation may require maintenance mode and a host reboot, which means VMs must be migrated first.

6.5 vSAN Troubleshooting (Objective 5.5)

vSAN troubleshooting is one of the most heavily tested areas on the exam because VCF relies entirely on vSAN for principal storage in the management domain. You must understand disk group architecture, object placement, storage policies, and the health monitoring tools.

Disk Failures and Rebuild Operations

When a capacity disk fails in a vSAN disk group, vSAN automatically begins rebuilding affected object components on remaining healthy disks across the cluster, assuming sufficient capacity exists and the storage policy can still be satisfied. If a cache disk fails, the entire disk group is marked as failed because all capacity disks in that group depend on the cache tier. This is a critical distinction — a single cache disk failure takes the entire disk group offline. Monitor rebuild progress through the vSAN health service or CLI:

# On ESXi host — check vSAN disk health
esxcli vsan health cluster list
esxcli vsan storage list

# Check resync status (rebuild progress)
esxcli vsan debug resync summary

# Via RVC (Ruby vSphere Console)
vsan.health.get /datacenter/computers/cluster-name

Network Partition (Split-Brain)

A vSAN network partition occurs when the vSAN VMkernel network is disrupted, splitting the cluster into two or more partitions that cannot communicate. Objects that have components on both sides of the partition lose quorum and become inaccessible. This is distinct from a simple host failure — a partition means hosts are running but cannot coordinate. In stretched cluster configurations, the vSAN witness host acts as a tiebreaker to maintain quorum on the preferred site. If witness connectivity is lost in a stretched cluster, the preferred fault domain retains quorum for objects whose policy allows it.

Object States

Understanding vSAN object health states is critical for the exam. Every VM's files (VMDK, VMX swap, namespace) are stored as vSAN objects composed of multiple components distributed across hosts:

  • Healthy: All components are accessible and the storage policy is fully met. The object can tolerate the number of failures defined by its FTT setting.
  • Degraded: The object is accessible, but one or more components are unavailable. The policy (e.g., FTT=1) can still tolerate the current failures, but no additional failures can be absorbed. Immediate attention is recommended.
  • Absent: Components are temporarily unavailable (e.g., a host is rebooting or in maintenance mode). vSAN waits 60 minutes (the "absent" timer) by default before starting a full rebuild to a new location. This timer prevents unnecessary data movement during planned maintenance.
  • Inaccessible: The object cannot be read or written because quorum is lost. This is a critical state — the VM cannot run. Immediate remediation is required, typically by restoring the failed host or its storage.

Exam Trap: vSAN "ensure accessibility" during host decommission means data cannot be fully evacuated to other hosts. This mode only moves data that would become inaccessible if the host were removed. If the decommission fails even with "ensure accessibility," it means remaining cluster capacity or host count cannot satisfy FTT requirements. This is a capacity and policy problem — check remaining disk space and whether the cluster can still meet the storage policy with one fewer host. Do not look at networking or DNS.

Key vSAN Tools and Logs

The vSAN health service provides a comprehensive health dashboard accessible through the vSphere Client under Cluster > Monitor > vSAN > Health. For CLI-based troubleshooting, use the Ruby vSphere Console (RVC) with the vsan.health.get command for a quick health summary and vsan.check_state for object state analysis. Key log files include:

  • /var/log/vsantraced/ — vSAN data path trace logs
  • /var/log/vobd.log — vSAN observer daemon logs
  • /var/log/vmkernel.log — kernel-level vSAN events and storage errors
  • vSAN performance service data — accessible via the vSphere Client performance graphs

6.6 NSX Troubleshooting (Objective 5.6)

NSX provides the entire network virtualization layer in VCF. Troubleshooting NSX requires understanding three distinct planes: the management plane (NSX Manager cluster), the control plane (Central Control Plane on NSX Manager), and the data plane (transport nodes, TEPs, Edge VMs).

Routing Issues

BGP Neighbor Down: If a Tier-0 gateway cannot establish a BGP session with a physical router, troubleshoot in this order: (1) Verify IP connectivity between the Edge uplink interface and the physical router — can you ping the peer IP? (2) Confirm the BGP AS numbers match on both sides. (3) Check that BGP timers and MD5 authentication passwords match. (4) Verify the physical router's BGP configuration allows the session from the Edge IP. Check BGP status from the Edge node:

# From NSX Edge CLI
get logical-routers
# Note the VRF ID for the Tier-0 SR (Service Router)
vrf <vrf-id>
get bgp neighbor summary
get route

Static Route Missing: If traffic is not reaching a destination network, verify that static routes are configured on the correct gateway tier (Tier-0 for external routes, Tier-1 for internal routes) and that route redistribution between Tier-0 and Tier-1 is enabled. A common exam scenario involves a newly created Tier-1 connected network that is unreachable from outside — the fix is enabling route advertisement on the Tier-1 gateway for "Connected Subnets."

Distributed Firewall (DFW) Rule Not Applying

The Distributed Firewall is enforced at the vNIC level on every workload VM. When a DFW rule appears to have no effect, follow this troubleshooting sequence:

  1. Check the Applied-To field — this is the number one cause of DFW rules not taking effect. If "Applied To" is set to a specific security group that does not include the target VMs, the rule is never evaluated for those VMs. Setting Applied-To to "DFW" applies the rule globally across all workloads.
  2. Check rule order — DFW rules are processed top-to-bottom within each category. A broader "Allow Any" rule higher in the list will match before your specific deny rule lower in the list.
  3. Verify the rule is published — unpublished rules in draft state have no effect on the data plane.
  4. Check group membership — if the rule references a dynamic security group, verify the target VM is actually a member. Check under Inventory > Groups > View Members in NSX Manager.
  5. Verify category precedence — remember that the Emergency category is processed before Application. A rule in Emergency can override anything in Application.

Key Concept: DFW Rule Processing Order

DFW rules are evaluated in strict category order: Ethernet → Emergency → Infrastructure → Environment → Application. Within each category, rules are processed top-to-bottom, and the first matching rule wins (no further rules are evaluated for that packet). The default rule at the bottom of the Application category is typically "Allow" in greenfield VCF deployments. Changing this to "Drop" implements a default-deny posture that blocks all traffic not explicitly permitted by higher rules.

Edge VM Issues

NSX Edge VMs provide all north-south services: routing, NAT, VPN, load balancing, and DNS forwarding. Common Edge issues include:

  • Deployment failures: Often caused by insufficient CPU/memory on the target host, incorrect network port group mappings, or datastore space limitations. Edge VMs come in multiple form factors (small, medium, large, extra-large) — ensure the target host can accommodate the selected size.
  • Connectivity loss on uplinks: Verify the Edge VM's uplink interface is connected to the correct VLAN-backed port group and that the physical switch port is configured as a trunk carrying the required VLANs.
  • Resource contention: Edge VMs handling high traffic volumes (especially with NAT or load balancing) can become CPU-bound. In production VCF deployments, Edge VMs should run on dedicated Edge cluster hosts with CPU and memory reservations.

Transport Node Connectivity

Transport nodes (ESXi hosts prepared for NSX) communicate overlay traffic via Tunnel Endpoints (TEPs). Each transport node has one or more TEP interfaces (typically vmk10, vmk11) that encapsulate overlay frames in Geneve tunnels. If TEP-to-TEP communication fails, all overlay segment traffic between those hosts is disrupted while VLAN-backed traffic continues to work normally. Verify TEP connectivity:

# From ESXi host — test TEP connectivity with jumbo frames
vmkping -I vmk10 -s 8972 -d <remote-TEP-IP>

# Check transport node tunnel status
esxcli network ip neighbor list   # Verify ARP entries for TEPs

Check transport node status in NSX Manager under System > Fabric > Nodes > Host Transport Nodes. A status of "Success" means full connectivity. "Degraded" indicates partial tunnel failures. "Failed" means the NSX modules are not installed or running on the host.

6.7 vSphere Supervisor Troubleshooting (Objective 5.9)

The vSphere Supervisor (formerly Workload Management) enables Kubernetes workloads to run natively on vSphere infrastructure. Troubleshooting Supervisor issues requires understanding the control plane VM architecture and its dependencies on networking, storage, and content libraries.

Supervisor Cluster Stuck in "Configuring" State

This is one of the most commonly tested Supervisor issues. The Supervisor deployment creates three control plane VMs that form a Kubernetes control plane. If any of these VMs fails to deploy, fails to boot, or cannot establish cluster communication, the Supervisor remains in "Configuring" state indefinitely. Systematic troubleshooting approach:

  • Network configuration errors: The management network, workload network, and ingress/egress CIDRs must be correctly configured and routable. Overlapping CIDR ranges between management and workload networks are a common misconfiguration.
  • Insufficient resources: Each control plane VM requires dedicated CPU and memory. If the cluster is heavily utilized, the control plane VMs may not receive their resource reservations, causing startup failures.
  • DNS resolution: Control plane VMs must resolve each other and the vCenter FQDN by hostname. DNS failures here cause the Kubernetes API server to fail initialization.
  • Content library sync failure: The Supervisor relies on a subscribed content library for the Kubernetes node images. If the content library cannot sync from the VMware source (due to proxy settings, firewall rules, or DNS issues), the control plane VM OVAs cannot be downloaded.
  • Load balancer issues: If using NSX Advanced Load Balancer (Avi) or HAProxy, verify the load balancer is reachable and has available capacity for the Supervisor virtual IP.

Control Plane VM Issues

If the Supervisor is enabled but a control plane VM becomes unresponsive, check its health using kubectl from the Supervisor context or by SSH-ing directly to the control plane VM:

# From a workstation with kubectl configured for the Supervisor
kubectl get nodes
kubectl get pods -n kube-system
kubectl get pods -n vmware-system-tkg

# From control plane VM (SSH as root)
systemctl status kubelet
crictl ps    # Check running containers
journalctl -u kubelet --since "1 hour ago"

Namespace and Resource Quota Issues

When developers report that pods are stuck in "Pending" state within a vSphere Namespace, check resource quotas first. If the namespace has reached its CPU, memory, or storage limit, the Kubernetes scheduler cannot place new pods. Adjust quotas through the vSphere Client under Workload Management > Namespaces > select namespace > Capacity and Usage. Also verify that storage policies assigned to the namespace are satisfied by the underlying vSAN configuration.

Harbor Registry and Network Policy Problems

The embedded Harbor registry depends on Supervisor networking to serve container images. If Harbor is unreachable, image pulls fail and pods remain in "ImagePullBackOff" state. Verify that the Harbor service is running (kubectl get pods -n tanzu-system-registry) and that workload network routing allows pods to reach the Harbor endpoint. For network policy issues, remember that vSphere Namespace network policies use the Kubernetes NetworkPolicy API — overly restrictive policies can block legitimate pod-to-pod or pod-to-registry traffic.

6.8 VCF Automation, VPC, Organizations, and HCX (Objectives 5.10–5.11)

VCF Automation Deployment Failures

VCF Automation (formerly vRealize Automation / Aria Automation) is deployed through SDDC Manager lifecycle management. Deployment failures are commonly caused by insufficient cluster resources (the Automation appliance requires significant CPU and memory), network connectivity failures between SDDC Manager and the target cluster, DNS resolution issues for the Automation FQDN, or certificate trust chain problems. Check SDDC Manager LCM logs at /var/log/vmware/vcf/lcm/ for deployment task errors. Post-deployment, the Automation appliance logs are in /var/log/vmware/vco/ (orchestrator) and /var/log/vmware/vra/ (automation engine).

VPC Networking Issues

VCF VPC (Virtual Private Cloud) provides network isolation for tenants using NSX constructs. Common troubleshooting scenarios include:

  • Incorrect subnet configuration: Overlapping CIDR blocks between VPCs or between a VPC and the management network cause routing conflicts.
  • Missing route advertisements: If a VPC workload cannot reach external networks, verify the Tier-1 gateway associated with the VPC has route advertisement enabled and is connected to the Tier-0 gateway. Check that NAT rules (SNAT for outbound, DNAT for inbound) are correctly configured.
  • DHCP scope conflicts: VPC subnets using DHCP may experience address exhaustion or conflicts if the scope is too small or overlaps with static assignments.

Organization Provisioning Problems

VCF Organizations provide multi-tenancy at the platform level. Provisioning failures typically relate to identity source integration (Active Directory or LDAP connectivity issues, incorrect bind credentials), resource allocation errors (insufficient capacity in the designated resource pool), or network pool exhaustion (no available VLANs or IP ranges for the new organization). Verify that sufficient VLANs, IP pools, and compute resources are available before provisioning.

HCX Migration Failures

VMware HCX enables workload migration between sites. The three main migration types each have distinct failure modes that are commonly tested:

  • Network Extension Issues: If the Layer 2 network extension between sites fails, verify that the HCX Network Extension (NE) appliance pair has established connectivity. Check that the source and destination port groups are correctly mapped, that the uplink network between sites permits the required traffic (UDP 4500 for IPSec, or direct connectivity), and that the NE appliance has sufficient throughput capacity.
  • Bulk Migration (Replication) Failures: Bulk migration uses host-based replication to transfer VM data. Failures are commonly caused by insufficient WAN bandwidth (causing replication timeouts), storage capacity exhaustion at the destination, incompatible VM hardware versions, or replication timeout settings that are too aggressive for the available bandwidth.
  • vMotion Migration Failures: HCX vMotion requires low latency between sites (under 150ms RTT). High latency or packet loss will cause vMotion migrations to fail or stall at high percentage completion. For large VMs, consider bulk migration instead of vMotion when latency is a concern.
  • Service Mesh Issues: The HCX Service Mesh connects the source and destination sites. If the service mesh shows "Degraded" status, check that all HCX appliances (Interconnect, Network Extension, WAN Optimization) are powered on and that their management and uplink interfaces have connectivity.

Access HCX logs through the HCX Manager appliance at /opt/vmware/var/log/hcx/ and through the HCX Manager UI under System > Logs.

6.9 The "Given a Scenario" Strategy

The majority of troubleshooting questions on the VCP-vCF exam are presented as scenario-based questions beginning with "An administrator reports that..." or "A user is unable to..." These questions test your ability to systematically diagnose problems rather than simply recall facts. Use this structured four-step approach to work through every troubleshooting question.

Step 1: Identify the Symptoms

Read the scenario carefully and extract every symptom described. Note what is working and what is not — both pieces of information are valuable. For example: "VMs on Host-A can communicate with VMs on Host-B, but not with VMs on Host-C, and all three hosts are on the same overlay segment." This tells you overlay networking works for some host pairs but not others, strongly suggesting a TEP connectivity issue specific to Host-C rather than a global NSX or segment configuration problem.

Step 2: Narrow to the Component

Based on the symptoms, identify which VCF component is most likely involved. Ask yourself: Is this a compute issue (vSphere/ESXi)? A storage issue (vSAN)? A networking issue (NSX)? A management issue (vCenter/SDDC Manager)? The symptoms guide you directly. Storage I/O errors and VM disk latency point to vSAN. Routing failures and firewall blocks point to NSX. VM power-on failures and migration errors point to HA, DRS, or resource constraints.

Step 3: Check the Appropriate Log

The exam frequently asks which log file you would check first or which CLI command you would run. Matching the right log to the right component is a directly testable skill. Memorize these mappings:

  • vCenter core issues → /var/log/vmware/vpxd/vpxd.log
  • ESXi host issues → /var/log/hostd.log, /var/log/vmkernel.log
  • HA failover issues → /var/log/fdm.log
  • DRS placement issues → /var/log/vmware/vpxd/vpxd.log (DRS runs on vCenter)
  • vSAN issues → /var/log/vsantraced/, vSAN health service, RVC
  • NSX Manager issues → /var/log/proton/nsxapi.log
  • NSX Edge issues → /var/log/syslog on the Edge VM
  • VCF bring-up issues → /opt/vmware/bringup/logs/ on Cloud Builder
  • VCF lifecycle/upgrade issues → /var/log/vmware/vcf/lcm/ on SDDC Manager

Step 4: Select the Most Likely Root Cause

With the component and log identified, evaluate the answer choices against your analysis. The correct answer will align with the component you identified and reference an appropriate remediation for that component. If your analysis points to NSX but all NSX-related answers seem wrong, reconsider your component identification — you may have missed a symptom in the scenario.

The Elimination Technique

For every troubleshooting question, you can typically eliminate at least two answers immediately using these principles:

  1. Wrong component: If the scenario clearly describes a networking symptom (e.g., "cannot ping"), eliminate answers about storage configuration or vSAN policies.
  2. Wrong log file: If an answer suggests checking fdm.log for a vSAN capacity problem, eliminate it immediately — fdm.log is the HA agent log, not a vSAN log.
  3. Impossible state: If an answer references a feature that does not exist in the described version, a configuration option that is not real, or a state that is logically impossible given the scenario, eliminate it.
  4. "Skip" or "ignore" answers: Any answer that suggests skipping a pre-check, ignoring a health warning, disabling a safety feature, or bypassing validation is almost always wrong. VMware designs these safeguards intentionally.

Key Concepts Summary

  • DNS is king: Both forward (A) and reverse (PTR) DNS records are required for every VCF component. DNS failures are the single most common bring-up issue and a favorite exam topic.
  • Pre-checks are mandatory: VCF upgrade pre-check failures must be resolved before proceeding. Skipping pre-checks is never the correct answer.
  • HA isolation response depends on heartbeat datastores: The isolation response behavior changes depending on whether heartbeat datastores are configured and responding. Network-only heartbeat failure with successful datastore heartbeats means partition, not isolation.
  • vSAN object states matter: Know the exact difference between degraded (accessible but reduced tolerance), absent (temporarily unavailable with 60-minute rebuild timer), and inaccessible (quorum lost, VM cannot run).
  • NSX DFW Applied-To field: The most common cause of firewall rules not applying is an incorrect scope in the Applied-To field. Always check this first.
  • TEP connectivity requires jumbo frames: Overlay network traffic between transport nodes uses Geneve-encapsulated TEP-to-TEP tunnels. MTU 9000 must be configured end-to-end on the physical network. Standard MTU will cause silent packet drops.
  • Supervisor "Configuring" state: Usually caused by network misconfiguration, DNS failure, content library sync failure, or insufficient resources for the three control plane VMs.
  • Log file matching is testable: Matching the correct log file to the correct component and failure type is a directly testable skill. The log path tables in this chapter are exam-critical reference material.
  • Cache disk failure takes the entire disk group offline: Unlike a capacity disk failure (which affects only the data on that disk), a cache disk failure disables all capacity disks in the same disk group.
  • DFW rule category order: Ethernet, Emergency, Infrastructure, Environment, Application — processed in that fixed order, top-to-bottom within each category, first match wins.

Exam Tips — Five Critical Traps in Troubleshooting Questions

  1. DNS Forward AND Reverse: When a deployment fails at DNS, always check BOTH forward AND reverse records. The exam will present scenarios where forward lookup succeeds but the reverse PTR record is missing or points to an incorrect hostname. Both record types must exist and must match each other. A working nslookup <hostname> does not mean DNS is fully configured — you must also verify nslookup <IP-address> returns the correct FQDN.
  2. vSAN "Ensure Accessibility" During Decommission: When decommissioning a host and selecting "Ensure accessibility" as the data evacuation mode, you are instructing vSAN to move only the data that would become inaccessible without this host. If the decommission task fails even with this mode, it means the remaining cluster cannot maintain object accessibility — there is not enough capacity or not enough hosts to satisfy the FTT policy requirements. This is a capacity and policy problem. Check remaining disk space and host count relative to the storage policy. Do not look at networking, DNS, or vCenter connectivity — those are distractors.
  3. NSX DFW Rules Not Working — Check Applied-To First: When a Distributed Firewall rule appears to have no effect, the very first thing to verify is the Applied-To field. If the rule's scope does not include the VMs you are trying to protect, the rule is never evaluated for those VMs, regardless of how the source, destination, and service fields are configured. Wrong scope is the number one cause of DFW rules not applying. Only after confirming Applied-To is correct should you investigate rule ordering, group membership, or publication status.
  4. HA Isolation Response Depends on Heartbeat Datastores: When the exam asks about HA isolation response behavior, the correct answer depends entirely on whether heartbeat datastores are configured. With heartbeat datastores, a host that loses network heartbeats will check datastore heartbeats before declaring itself isolated — if datastore heartbeats succeed, the host is partitioned (not isolated) and VMs continue running. Without heartbeat datastores, isolation is declared based solely on network heartbeat failure, and the configured isolation response executes. Read the scenario carefully for mentions of heartbeat datastore configuration.
  5. Upgrade Pre-Check Failures — Never Skip: "Skip pre-check" or "ignore the warning" is NEVER the correct answer on the exam. When a pre-check fails, you must resolve the underlying issue — patch a component to a compatible version, fix a health alarm, free cluster resources, or rotate expired credentials — and then re-run the pre-check successfully. VMware's upgrade orchestration is designed so that pre-checks catch problems that would cause catastrophic failures mid-upgrade. Bypassing them is not a supported or recommended action.

Architecture (Questions 1–4)

Question 1

A VCF administrator is planning a new VCF 9.0 deployment for a global enterprise with data centers in New York, London, and Tokyo. Each data center needs independent lifecycle management, but the organization wants centralized health monitoring and cross-site workload mobility. How should the administrator structure the VCF deployment?

A. Deploy a single VCF Instance with three Workload Domains, one per data center

B. Deploy three separate VCF Fleets, one per data center, with manual coordination

C. Deploy a single VCF Fleet with three VCF Instances, one per data center

D. Deploy a single VCF Instance with a stretched Management Domain across all three sites

Correct Answer: C

C is correct because a Fleet provides centralized management across multiple Instances, each Instance can operate independently with its own lifecycle, and Fleet-level operations enable cross-instance workload mobility. A is wrong because Workload Domains within a single Instance share the same Management Domain — they don't provide geographic independence. B is wrong because separate Fleets cannot share centralized monitoring or workload mobility. D is wrong because stretching a Management Domain across three geographies is not a supported configuration.

An administrator needs to deploy the initial Management Domain for a new VCF 9.0 instance. Which tool should they use?

A. Cloud Builder OVA appliance

B. VCF Operations deployment wizard

C. VCF Installer container

D. SDDC Manager bring-up workflow

Correct Answer: C

C is correct because VCF 9.0 uses VCF Installer (a container) to perform Management Domain bring-up. A is wrong because Cloud Builder was the previous-generation tool (deployed as a VM appliance), replaced in VCF 9.0. B is wrong because VCF Operations manages day-2 operations after deployment, not initial bring-up. D is wrong because SDDC Manager has been replaced by VCF Operations in VCF 9.0.

Question 3

A company needs to run Kubernetes workloads in a production VCF environment with maximum resilience against entire failure domain outages. Which Supervisor deployment mode should the administrator configure?

A. Single Host Supervisor

B. Easy Supervisor

C. Single-Zone Supervisor

D. Multi-Zone Supervisor

Correct Answer: D

D is correct because Multi-Zone Supervisor distributes control plane VMs across multiple vSphere Zones (failure domains/AZs), surviving an entire zone failure. A is wrong because Single Host has zero redundancy and is for dev/test only. B is wrong because Easy Supervisor simplifies deployment but doesn't provide cross-AZ resilience. C is wrong because Single-Zone runs within a single failure domain — a zone failure takes down the Supervisor.

Question 4

An architect is designing NSX connectivity for a VCF environment with 12 workload domains serving different business units. Each business unit requires independent routing policies and firewall rules. What NSX connectivity model should be used?

A. Centralized connectivity with a shared Tier-0 gateway

B. Distributed connectivity with per-domain Tier-0 gateways

C. Centralized connectivity with per-domain Tier-1 gateways only

D. No NSX gateway — use distributed firewall only

Correct Answer: B

B is correct because distributed connectivity gives each workload domain its own Tier-0 gateway, enabling independent routing policies, firewall rules, and BGP peering per business unit. A is wrong because centralized connectivity shares a single Tier-0 — all domains share routing and firewall policies at the T0 level. C is wrong because T1-only cannot provide independent north-south routing policies. D is wrong because DFW handles east-west micro-segmentation, not north-south routing.

Question 7

An administrator is commissioning new ESXi hosts into VCF. The validation check fails with a DNS error. Which specific DNS issue is most likely the cause?

B. Only forward DNS records exist — reverse records are missing

Question 10

An administrator needs to upgrade NSX in their VCF 9.0 environment to address a critical security vulnerability. What is the correct approach?

A. Use the NSX upgrade coordinator to upgrade NSX Manager and Edge nodes directly

Question 11

After deploying a new VCF 9.0 workload domain with NSX, an administrator creates Gateway Firewall rules on the Tier-1 gateway but reports that traffic is not being filtered. What is the most likely cause?

A. The Tier-1 gateway does not support Gateway Firewall

B. The firewall rules are in the wrong order

C. Gateway Firewall is disabled by default in NSX 9.0 greenfield deployments

D. The Edge cluster needs additional Edge nodes for firewall processing

Correct Answer: C

C is correct because in NSX 9.0 greenfield deployments, Gateway Firewall is DISABLED by default on both T0 and T1 gateways. The administrator must explicitly enable it before rules will take effect. A is wrong because T1 does support Gateway Firewall. B is wrong because rule order doesn't matter if the firewall service itself is disabled. D is wrong because Edge node count doesn't affect whether Gateway Firewall is enabled.

Question 12

An administrator is planning ESXi host upgrades for a VCF workload domain that runs critical NSX Edge VMs. The Edge VMs cannot be vMotioned due to SR-IOV passthrough configuration. How should the administrator handle the Edge VMs during the upgrade?

A. Remove the SR-IOV passthrough, vMotion the Edge VMs, then re-add passthrough

B. Configure Edge Host Affinity to fail over Edge VMs without vMotion

C. Power off the Edge VMs and accept network downtime during the upgrade

D. Skip upgrading hosts running Edge VMs

Correct Answer: B

B is correct because Edge Host Affinity enables Edge VM failover without vMotion. The Edge VM is shut down on the source host and restarted on a designated affinity host, while NSX Edge HA handles service failover. A is wrong because reconfiguring SR-IOV during an upgrade is complex and error-prone. C is wrong because network downtime is unacceptable for critical Edge VMs. D is wrong because all hosts must be upgraded to maintain BOM compliance.

Question 13

An NSX administrator notices that NSX Manager can no longer communicate with several Transport Nodes. The TN certificates are 820 days old. What tool should be used to resolve this?

A. Re-deploy the Transport Nodes with new certificates

B. Run the CARR script to renew Transport Node certificates

C. Replace the NSX Manager cluster certificates

D. Reset the Transport Node agent on each affected host

Correct Answer: B

B is correct because the CARR (Certificate Authority Renewal and Rotation) script is specifically designed for Transport Node certificate lifecycle management. At 820 days, the certificates are approaching the 825-day threshold and need renewal. A is wrong because re-deploying TNs is destructive and unnecessary. C is wrong because the issue is TN certificates, not NSX Manager certificates. D is wrong because resetting the agent doesn't renew expired certificates.

Question 14

Overlay network traffic between ESXi hosts in a VCF workload domain is experiencing packet drops. The physical switches are configured with the default MTU of 1500. What is the most likely cause?

A. The NSX Manager cluster is overloaded

B. TEP IP addresses are exhausted in the pool

C. The MTU is too low — overlay traffic requires minimum 1600 bytes

D. The DFW is blocking inter-host overlay traffic

Correct Answer: C

C is correct because Geneve encapsulation adds overhead to packets. The minimum MTU for NSX overlay networks is 1600 bytes. With physical switches at 1500, encapsulated packets are being fragmented or dropped. A is wrong because NSX Manager handles the control plane, not data plane traffic. B is wrong because TEP IP exhaustion prevents new TN configuration, not packet drops. D is wrong because DFW operates at the VM level, not on encapsulated overlay traffic between hosts.

Storage (Questions 15–18)

Question 15

An administrator is planning storage for a new VCF 9.0 workload domain. The available hardware has NVMe drives only. Which vSAN architecture should be used and why?

A. OSA — because it supports all drive types including NVMe

B. ESA — because it is designed for NVMe-only configurations with a single storage tier

C. OSA — because NVMe drives should be used as cache tier devices

D. Either ESA or OSA — both work identically with NVMe drives

Correct Answer: B

B is correct because ESA (Express Storage Architecture) is designed specifically for NVMe-only environments. It uses a single flat storage tier (no cache/capacity split), delivers better performance, and is the VCF 9.0 default for new deployments. A and C are wrong because while OSA supports NVMe, it wastes NVMe performance by forcing the cache/capacity tier distinction. Question 16

A VCF stretched cluster spans two data centers. The round-trip latency between data sites is 3ms. The witness host is located in a cloud provider with 180ms RTT to each data site. Is this configuration supported?

B. No — the witness latency exceeds the 100ms maximum

C. Yes — both latencies are within the supported limits

D. No — the witness must be on-premises, not in a cloud provider

Correct Answer: C

C is correct because the limits are ≤5ms RTT between data sites (3ms ✓) and ≤200ms RTT to witness (180ms ✓). Both are within spec. A is wrong because the limit is 5ms, not 1ms. Question 17

An administrator needs to enable vSAN encryption for data stored on disk but the organization does not have a KMS server. What are their options?

C. Enable both data-at-rest and data-in-transit encryption using vCenter as the key server

Question 18

A vSAN cluster has 6 hosts with 10 TB raw capacity each (60 TB total). The storage policy is FTT=1 with RAID-5 erasure coding. How much usable capacity is available after accounting for 25% slack space?

A. 15 TB

B. 22.5 TB

C. 33.75 TB

D. 45 TB

Correct Answer: C

C is correct. Calculation: 60 TB raw × 75% (RAID-5 FTT=1 efficiency) = 45 TB effective. 45 TB × 75% (minus 25% slack) = 33.75 TB usable. A uses RAID-1 math (50%) with slack. B forgets to account for RAID overhead correctly. D forgets slack space entirely.

Troubleshooting (Questions 23–25)

Question 23

An administrator attempts to decommission an ESXi host from a 4-node vSAN cluster with FTT=1 (RAID-1). The decommission fails with the error "Cannot ensure data accessibility." The vSAN datastore is 85% full. What is the root cause?

A. The host has VMs that cannot be vMotioned

B. The vSAN cluster does not have enough free capacity to rebuild objects on the remaining 3 hosts while maintaining FTT=1

C. The host has not been placed in maintenance mode first

D. FTT=1 requires a minimum of 5 hosts and cannot run on 3

Correct Answer: B

B is correct. At 85% full on a 4-node cluster, removing one host means the remaining 3 hosts must absorb all data from the departing host while maintaining FTT=1 (2 copies of every object). With only 15% free space, there isn't enough capacity. The fix is to add storage/hosts or reduce FTT before decommissioning. A is wrong because VM migration is a separate step from data evacuation. C is wrong because maintenance mode is part of the decommission workflow — the error is about data, not process order. D is wrong because FTT=1 with RAID-1 requires minimum 3 hosts.

Question 24

An administrator creates a new NSX Distributed Firewall policy to block SSH traffic between two application tiers in a VCF workload domain. After publishing the policy, SSH traffic continues to flow between the tiers. The rule syntax is correct and uses the right IP addresses. What should the administrator check FIRST?

A. Whether the NSX Manager cluster is healthy

B. The "Applied To" field on the DFW rule to verify it targets the correct scope

C. Whether the ESXi hosts have the NSX VIBs installed

D. The Edge cluster firewall logs for denied traffic

Correct Answer: B

B is correct because the #1 cause of correctly-configured DFW rules not taking effect is the "Applied To" field being set to the wrong scope. The rule may target "DFW" (all VMs) when it should target a specific group, or vice versa. A is wrong because NSX Manager health wouldn't explain a published-but-ineffective rule. C is wrong because if VIBs were missing, no DFW rules would work — not just this one. D is wrong because DFW operates on hosts, not Edge nodes — Edge firewall logs are irrelevant for distributed firewall rules.

Question 25

During a VCF upgrade, the pre-check validation fails with the finding: "Certificate for component NSX-Manager-1 expires in 15 days." The administrator wants to proceed with the upgrade quickly. What is the correct action?

A. Skip the pre-check and proceed with the upgrade — the certificate won't expire during the upgrade window

B. Override the pre-check warning since it's not a critical failure

C. Renew the NSX Manager certificate, then re-run the pre-check validation before proceeding

D. Proceed with the upgrade — the upgrade process will automatically renew expiring certificates

Correct Answer: C

C is correct because pre-check failures MUST be resolved before proceeding with an upgrade. The certificate must be renewed first, then pre-checks re-run to validate the fix. A is wrong because "skip pre-check" is NEVER the correct answer. B is wrong because pre-check findings cannot be overridden. D is wrong because the upgrade process does not automatically renew expiring certificates — and if the cert expires mid-upgrade, it could cause a catastrophic failure.
